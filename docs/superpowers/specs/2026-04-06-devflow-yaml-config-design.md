# Design: Configuração YAML do DevFlow (`.context/.devflow.yaml`)

**Data:** 2026-04-06
**Escopo:** Hook git-strategy configurável via YAML + entrevista `/devflow config`

---

## Problema

O hook `pre-tool-use` do DevFlow detecta a estratégia git por heurísticas frágeis (grep em Markdown, detecção de branches/scripts). Não existe um arquivo de configuração padronizado que o desenvolvedor possa editar. Além disso, comportamentos automáticos como auto-finish (bump, commit, push, merge) não são configuráveis.

## Decisões

| Decisão | Escolha |
|---------|---------|
| Local do arquivo | `.context/.devflow.yaml` |
| Quando criar | `/devflow init` (novo) ou `/devflow config` (existente) |
| Estratégias suportadas | branch-flow, worktree, trunk-based (3) |
| Hook sem YAML | Bloqueia + instrui `/devflow config` |
| Source of truth | Apenas o YAML (sem fallback para heurísticas) |
| Heurísticas | Usadas no `/devflow config` para pré-selecionar recomendação |
| `autoFinish` ausente | Desativado (opt-in) |
| `branchProtection` | Existe, mas trunk-based ignora (sempre false) |

---

## Schema do `.context/.devflow.yaml`

```yaml
# .context/.devflow.yaml — DevFlow project configuration
# Generated by: /devflow init or /devflow config

git:
  strategy: branch-flow              # branch-flow | worktree | trunk-based
  protectedBranches: [main, develop] # branches que o hook bloqueia
  prCli: gh                          # gh | glab | none
  branchProtection: true             # false = hook não bloqueia edições

  # Auto-finish: ausente = desativado (opt-in)
  # Boolean simples:
  # autoFinish: true                 # ativa tudo (bump + commit + push + merge)
  # autoFinish: false                # desativa tudo (explícito)
  #
  # Granular (chaves ausentes = false):
  # autoFinish:
  #   bump: true
  #   commit: true
  #   push: false
  #   merge: false
```

### Regras de validação

| Strategy | `branchProtection` | `protectedBranches` | `autoFinish` |
|----------|-------------------|---------------------|--------------|
| `branch-flow` | respeitado | obrigatório (min 1) | respeitado |
| `worktree` | respeitado | obrigatório (min 1) | respeitado |
| `trunk-based` | ignorado (sempre false) | ignorado (vazio) | ignorado (sempre false) |

### Parsing do `autoFinish`

- Ausente → tudo `false` (desativado)
- `true` → tudo `true` (bump, commit, push, merge)
- `false` → tudo `false` (explícito)
- Objeto `{ bump: true, push: false }` → chaves ausentes = `false`

## Hook `pre-tool-use` (novo comportamento)

### Fluxo

```
Edit/Write interceptado
  │
  ├─ Arquivo é exceção (.context/workflow/*, docs/*, etc.)? → ALLOW
  │
  ├─ .context/.devflow.yaml existe?
  │   ├─ NÃO → DENY + "Execute /devflow config para configurar"
  │   │
  │   └─ SIM → Ler YAML
  │       ├─ strategy: trunk-based → ALLOW
  │       ├─ branchProtection: false → ALLOW
  │       └─ branchProtection: true (ou ausente)
  │           ├─ Branch atual ∈ protectedBranches?
  │           │   ├─ SIM → DENY + invocar git-strategy skill
  │           │   └─ NÃO → ALLOW
```

### Mudanças

- Remove: grep no `development-workflow.md`
- Remove: defaults hardcoded (`main master develop`)
- Adiciona: check de existência do `.devflow.yaml`
- Adiciona: parsing YAML via `python3` (já usado para JSON)
- Adiciona: mensagem i18n `MSG_NO_CONFIG` (3 idiomas)

## Skill `git-strategy` (adaptações)

### O que muda

- Lê estratégia diretamente do `.context/.devflow.yaml`
- Se YAML não existe → instrui `/devflow config` (não pergunta)
- Remove lógica de detecção por heurísticas (migrada para `/devflow config`)
- Remove as 3 perguntas de configuração inicial (migradas para `/devflow config`)

### Fluxo (sem mudança funcional)

1. Hook bloqueia → instrui invocar skill
2. Skill lê `.devflow.yaml` → sabe a strategy
3. Pergunta tipo de branch (feature/fix/hotfix/release)
4. Cria isolamento conforme strategy (checkout -b ou worktree add)

## Entrevista `/devflow config`

### Heurísticas de detecção (pré-seleção)

```
1. Tem scripts/wt-create.sh? → recomenda worktree
2. Tem branch develop? → recomenda branch-flow (main + develop)
3. Só main? → recomenda branch-flow (só main)
4. Nenhum sinal → recomenda branch-flow (main)
5. gh instalado? → pré-seleciona gh como prCli
6. glab instalado? → pré-seleciona glab como prCli
```

### Perguntas

**P1: Estratégia git** (com heurística pré-selecionada)
```
"Qual estratégia git deste projeto?"
● Branch Flow (Recomendado — detectei branch develop)
○ Worktree
○ Trunk-based
```

**P2: Branches protegidas** (só se branch-flow ou worktree)
```
"Quais branches são protegidas?"
☑ main
☐ develop
```
Pré-seleciona conforme heurística.

**P3: CLI de PR**
```
"Qual CLI para criação de PRs?"
○ gh (GitHub)
○ glab (GitLab)
○ Nenhuma
```

**P4: Branch protection**
```
"Ativar proteção de branch? (hook bloqueia edição em branches protegidas)"
● Sim (Recomendado)
○ Não
```

**P5: Auto-finish**
```
"Ativar finalização automática de branch?"
○ Não (padrão)
○ Sim, tudo (bump + commit + push + merge)
○ Personalizar
```
Se "Personalizar":
```
"Quais etapas ativar?"
☐ bump (version bump)
☐ commit
☐ push
☐ merge
```

### Resultado

Gera `.context/.devflow.yaml` com as respostas do usuário.

## Hook `post-tool-use` (auto-finish)

### Comportamento

| `autoFinish` | Efeito |
|---|---|
| ausente / `false` | Não sugere nem executa nada |
| `true` | Executa bump + commit + push + merge |
| `{ bump: true, push: false }` | Executa só chaves `true`, pula o resto |

### Fluxo

```
1. Ler .context/.devflow.yaml → autoFinish
2. Se desativado → nada (usuário finaliza manualmente)
3. Se ativado (total ou parcial):
   - bump: true → executa version bump
   - commit: true → commit das mudanças
   - push: true → push para remote
   - merge: true → cria PR (via prCli) ou merge direto
4. Etapas desativadas → informa: "X desativado. Execute manualmente."
```

---

## Integração com `/devflow init`

O `devflow init` chama a mesma entrevista do `/devflow config` como parte do setup inicial. Se `.context/.devflow.yaml` já existe, o init pula essa etapa (usa o existente).
